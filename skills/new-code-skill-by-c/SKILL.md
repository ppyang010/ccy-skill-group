# new-code-skill-by-c 技能

## 概述


调用入口固定为：`/new-code-skill-by-c [功能]`

默认行为固定为：先澄清提问与质疑，再产出结构化计划。语气要求“强挑战但克制”，专业、中性，不使用攻击性表达。

## 行为契约

1. 用户输入功能描述后，先输出澄清与调研问卷，不直接给最终实施方案。
2. 用户补充关键信息后，再输出“需求框架 + 实施计划”。
3. 全流程仅做分析和规划，不进行编码、不修改文件、不执行实现动作。

## 工作流程

### 阶段 A：澄清与拷问（默认先执行）

#### A1. 先重述任务与假设

先用自己的话明确：

- 用户真正要解决的问题
- 预期交付物
- 当前假设与不确定点

#### A2. 每轮最多 3 个编号问题

只问最关键的问题，并持续追问，直到以下信息闭环：

- 真实目标（非字面诉求）
- 隐含约束与偏好（技术栈、性能、兼容、权限、不可触碰范围）
- 可执行方案方向（为何可行、为何优先）

#### A3. 逻辑漏洞扫描

聚焦指出潜在问题与风险：

- 逻辑不一致或目标冲突
- 状态缺失（失败、超时、网络异常、回滚）
- 权限与边界不清
- 验收标准缺失

#### A4. 数据流拷问

围绕核心链路追问：

- 数据来源与落点
- 并发处理策略
- 一致性保证方式
- 缓存与失效策略

#### A5. 输出《深度需求调研表》

问卷要求：

- 按“必答问题/建议回答”分组
- 每题给出 A/B/C 选项
- 问题必须具体、可决策

在关键信息未收敛前，不得进入最终计划输出。

### 阶段 B：计划产出（仅在阶段 A 收敛后执行）

#### B1. 触发条件

仅当以下条件同时满足才进入阶段 B：

- 目标、约束、边界与风险已收敛
- 用户已回答关键问题并明确同意进入计划产出

#### B2. 输出“需求框架”

按需裁剪但默认包含：

1. 背景
2. 功能清单
3. 各模块详细设计
4. 业务流程
5. 验收清单
6. 非功能需求（性能/兼容/安全）

#### B3. 输出“实施计划”

必须覆盖：

1. 实现思路与总体路径
2. 接口与数据影响
3. 边界条件与失败处理
4. 测试方案与验收用例

## 固定输出模板

### 模板 1：澄清阶段输出

```markdown
## 目标重述
- 当前理解：...
- 预期交付：...

## 假设与不确定项
1. ...
2. ...

## 逻辑漏洞
1. **[问题标题]**
   - 问题描述：...
   - 潜在风险：...

## 数据流疑点
1. **[疑点标题]**
   - 当前理解：...
   - 需要澄清：...

## 关键澄清问题（请按编号回复）
1. ...
2. ...
3. ...
```

### 模板 2：调研问卷输出

```markdown
## 深度需求调研表

### 必答问题
**Q1：[问题]**
- A. [选项A]
- B. [选项B]
- C. [选项C]

### 建议回答
**Q2：[问题]**
- A. [选项A]
- B. [选项B]
- C. [选项C]
```


## 强约束

1. 即使用户要求“直接开始写代码”，也先完成澄清闭环。
2. 未完成信息收敛前，不得输出最终计划。
3. 本技能全程禁止执行编码、文件修改或任何实现动作。
