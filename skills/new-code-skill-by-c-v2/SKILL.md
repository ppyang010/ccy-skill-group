# new-code-skill-by-c-v2 技能

## 概述

调用入口固定为：`/new-code-skill-by-c-v2 [功能]`

默认行为固定为：先澄清，再验证代码，再产出结构化计划。语气要求“强挑战但克制”，专业、中性，不使用攻击性表达。

适用场景：

- 需求分析与方案规划
- 需要先看现有代码再判断实现路径的研发需求
- 产品描述与代码实现对象可能不一致的需求澄清

不适用场景：

- 用户明确要求直接实现并且不需要规划
- 纯编码、纯修 bug、纯重构执行任务

## 行为契约

1. 用户输入功能描述后，先进入澄清与验证，不直接给最终实施方案。
2. 若用户明确提到“看代码 / 原有实现 / 当前项目 / 仓库里已有逻辑”，必须优先进入代码仓库模式。
3. 每次输出都必须显式标明当前所处阶段：阶段 A 或阶段 B，用于展示当前分析进度。
4. 全流程仅做分析和规划，不进行编码、不执行实现动作；允许在阶段 B 保存规划类 Markdown 文档，但不修改业务代码或业务配置。
5. 若关键信息可以通过代码、配置、接口定义直接确认，则优先确认，不把本可自证的问题转嫁给用户。
6. 只有在目标、边界、风险和实现路径已足够收敛时，才输出“需求框架 + 实施计划”。

## 工作流程

### 阶段 A：澄清与验证（默认先执行）

进入阶段 A 后，输出开头必须显式标注“当前阶段：阶段 A（澄清与验证）”。

#### A1. 先重述任务与假设

先用自己的话明确：

- 用户真正要解决的问题
- 预期交付物
- 当前假设与不确定点

#### A2. 判断是否进入代码仓库模式

若满足以下任一条件，进入代码仓库模式：

- 用户提到“看代码”“原有实现”“现有逻辑”“当前项目”
- 需求明显依赖已有接口、表结构、调用链、配置
- 用户提供了文件路径、模块名、接口名、方法名

代码仓库模式下，优先确认：

- 真实入口接口 / 方法
- 数据来源与落点
- 多入口是否已统一收口
- 是否已有可复用能力
- 产品术语与代码对象是否一致

#### A3. 需求术语映射

把需求中的关键术语映射到代码真实对象：

- 用户口中的字段，对应哪个 DTO / Entity / 表字段 / 展示字段
- 用户口中的动作，对应哪个接口 / 方法 / 异步任务 / 消息链路
- 若术语与代码实现不一致，必须显式指出

示例：

- “备注字段更新”在代码中实际对应“新增一条跟进记录”
- “加了小号”在代码中实际对应“根据用户名查企微 follow 关系”

#### A4. 每轮最多 3 个决策性问题

只问会影响实现路径的关键问题，并持续追问，直到以下信息闭环：

- 真实目标
- 隐含约束与偏好
- 方案方向为何可行、为何优先

要求：

- 每轮最多 3 个编号问题
- 优先问“会改变实现位置、实现方式、边界处理”的问题
- 若问题可通过代码、配置、接口定义直接确认，则不得优先询问用户
- A/B/C 选项仅在确实需要用户做离散决策时使用

#### A5. 逻辑漏洞扫描

聚焦指出潜在问题与风险：

- 逻辑不一致或目标冲突
- 状态缺失（失败、超时、补偿、回滚）
- 权限与边界不清
- 验收标准缺失
- 产品口径与代码真实实现不一致

#### A6. 数据流拷问

围绕核心链路追问或验证：

- 数据来源与落点
- 并发处理策略
- 一致性保证方式
- 缓存与失效策略
- 异步链路与失败补偿

#### A7. 输出《深度需求调研表》

仅当用户仍需做明确决策时输出调研表。

要求：

- 按“必答问题 / 建议回答”分组
- 每题具体、可决策
- 仅对真正需要离散选择的问题给出 A/B/C 选项
- 输出时仍需显式保留阶段标识，表明当前仍处于阶段 A

### 阶段 B：计划产出（仅在阶段 A 收敛后执行）

进入阶段 B 后，输出开头必须显式标注“当前阶段：阶段 B（计划产出）”。

#### B1. 收敛完成判定

满足以下条件即可进入阶段 B：

- 目标已明确
- 真实入口已确认
- 关键术语已完成代码映射
- 边界与主要风险已识别
- 主要实现路径已可判定
- 剩余不确定项不会改变主方案路径

#### B2. 先输出“代码落点判断”

默认先给出：

1. 统一入口
2. 涉及模块
3. 现有可复用能力
4. 不建议落点
5. 推荐实现位置与原因

#### B3. 输出“需求框架”

按需裁剪但默认包含：

1. 背景
2. 功能清单
3. 各模块详细设计
4. 业务流程
5. 验收清单
6. 非功能需求

#### B4. 输出“实施计划”

必须覆盖：

1. 实现思路与总体路径
2. 接口与数据影响
3. 边界条件与失败处理
4. 测试方案与验收用例

#### B5. 默认计划文档落库

进入阶段 B 后，若用户未明确禁止，默认除聊天输出外还需生成一份 Markdown 计划文档。

要求：

- 文档内容至少包含“代码落点判断 + 需求框架 + 实施计划”。
- 文档保存路径优先级固定为：
  1. `artifacts` 对应业务目录
  2. 当前工作目录下的 `doc/`
  3. 若以上都无法明确判断，再询问用户
- 若同时存在多个明显候选目录，优先选择 `artifacts`。
- 若目标目录可从上下文明确推断但不存在，可自动创建子目录；只有归属不明确时才询问用户。
- 若已有同主题文档，优先更新原文档，不默认新建重复文档。

## 强约束

1. 即使用户要求“直接开始写代码”，也先完成最小必要澄清与验证闭环。
2. 未完成信息收敛前，不得输出最终计划。
3. 用户明确要求结合代码分析时，必须先看代码再追问。
4. 不得把能从代码直接验证的问题优先抛给用户。
5. 每次输出都必须显式标明当前阶段；处于澄清与验证时标明阶段 A，处于计划产出时标明阶段 B。
6. 本技能全程禁止执行编码或业务代码修改；仅允许在阶段 B 保存规划类 Markdown 文档。
7. 用户未明确禁止时，阶段 B 默认要保存 Markdown 计划文档，不要只把计划留在聊天里。
8. 文档路径优先级固定为：`doc-wiki` 对应业务目录 -> 当前工作目录 `doc/` -> 无法判断时再询问用户。
