# code-design-task-planner 技能

## 概述

调用入口固定为：`/code-design-task-planner [功能]`

默认行为固定为：先澄清，再验证代码，再产出结构化计划和任务跟踪文档。语气要求“强挑战但克制”，专业、中性，不使用攻击性表达。

适用场景：

- 需求分析与方案规划
- 需要先看现有代码再判断实现路径的研发需求
- 产品描述与代码实现对象可能不一致的需求澄清

不适用场景：

- 用户明确要求直接实现并且不需要规划
- 纯编码、纯修 bug、纯重构执行任务

## 行为契约

1. 用户输入功能描述后，先进入澄清与验证，不直接给最终实施方案。
2. 若当前任务存在可访问的项目、仓库、模块、接口、路径或业务链路上下文，默认先查询已有代码、配置、接口定义和数据结构，再继续做术语映射与方案判断；只有用户明确禁止查代码，或当前任务确实不依赖现有代码时，才跳过代码验证并说明原因。
3. 每次输出都必须显式标明当前所处阶段：阶段 A 或阶段 B，用于展示当前分析进度。
4. 全流程仅做分析和规划，不进行编码、不执行实现动作；允许在阶段 B 保存规划类 Markdown 文档，但不修改业务代码或业务配置。
5. 若关键信息可以通过代码、配置、接口定义直接确认，则优先确认，不把本可自证的问题转嫁给用户。
6. 只有在目标、边界、风险和实现路径已足够收敛时，才输出“需求框架 + 实施计划”。
7. 需求讨论完成后，阶段 B 默认产出两份文档：`功能名-desgin.md` 和 `功能名-task.md`。文件名按用户要求沿用 `desgin` 拼写。
8. `功能名-task.md` 是后续执行的任务跟踪入口，任务状态必须随着执行进度同步变更。
9. 若需求目标、边界、接口、字段、数据结构、实现顺序、验收口径、任务优先级或范围出现变动，必须优先同步修改相关设计文档和任务文档，再继续推进后续动作。

## 工作流程

### 阶段 A：澄清与验证（默认先执行）

进入阶段 A 后，输出开头必须显式标注“当前阶段：阶段 A（澄清与验证）”。

#### A1. 先重述任务与收敛目标

先用自己的话明确：

- 用户真正要解决的问题
- 预期交付物
- 当前假设与不确定点
- 阶段 A 需要收敛到什么程度，才能进入阶段 B

#### A2. 代码仓库验证与术语映射

若当前任务存在可访问的项目上下文，默认先查询已有代码、配置、接口定义和数据结构，再完成术语映射；不要把“是否查代码”作为一轮显式提问优先抛给用户。

优先确认：

- 真实入口接口 / 方法
- 数据来源与落点
- 多入口是否已统一收口
- 是否已有可复用能力
- 产品术语与代码对象是否一致

术语映射时，优先落到以下真实代码对象：

- 用户口中的字段，对应哪个 DTO / Entity / 表字段 / 展示字段
- 用户口中的动作，对应哪个接口 / 方法 / 异步任务 / 消息链路
- 若术语与代码实现不一致，必须显式指出

示例：

- “备注字段更新”在代码中实际对应“新增一条跟进记录”
- “加了小号”在代码中实际对应“根据用户名查企微 follow 关系”

若当前任务不依赖现有代码、当前没有可定位仓库，或用户明确禁止查代码，则显式说明原因，并跳过代码验证。

本步骤默认产出：

- 已定位代码入口
- 术语映射结果
- 可复用能力
- 当前未确认点

#### A3. 风险与数据流验证

围绕核心链路验证潜在风险与实现边界，优先检查：

- 逻辑不一致或目标冲突
- 状态缺失（失败、超时、补偿、回滚）
- 权限与边界不清
- 验收标准缺失
- 数据来源与落点
- 并发处理策略
- 一致性保证方式
- 缓存与失效策略
- 异步链路与失败补偿

#### A4. 每轮最多 3 个决策性问题

基于 A2 和 A3 的验证结果，只问会影响实现路径的关键问题，并持续追问，直到以下信息闭环：

- 真实目标
- 隐含约束与偏好
- 方案方向为何可行、为何优先

要求：

- 每轮最多 3 个编号问题
- 优先问“会改变实现位置、实现方式、边界处理”的问题
- 若问题可通过代码、配置、接口定义直接确认，则不得优先询问用户
- A/B/C 选项仅在确实需要用户做离散决策时使用

#### A5. 输出《深度需求调研表》

仅当剩余决策较多、零散追问不利于继续收敛，或用户明确希望一次性列出待确认项时输出调研表。

要求：

- 按“必答问题 / 建议回答”分组
- 每题具体、可决策，且应来自当前仍未收敛的关键分歧
- 仅对真正需要离散选择的问题给出 A/B/C 选项
- 输出时仍需显式保留阶段标识，表明当前仍处于阶段 A

### 阶段 B：计划产出与文档落库（仅在阶段 A 收敛后执行）

进入阶段 B 后，输出开头必须显式标注“当前阶段：阶段 B（计划产出）”。

#### B1. 阶段 A 结论沉淀

满足以下条件即可进入阶段 B：

- 目标已明确
- 真实入口已确认
- 关键术语已完成代码映射
- 边界与主要风险已识别
- 主要实现路径已可判定
- 剩余不确定项不会改变主方案路径

进入阶段 B 后，默认先把阶段 A 的关键结论沉淀为设计基线，至少包括：

- 代码落点判断：统一入口、涉及模块、现有可复用能力、不建议落点、推荐实现位置与原因
- 需求术语映射结论
- 风险与边界结论
- 当前未确认点 / 暂不处理范围

`功能名-desgin.md` 用于沉淀上述设计基线，是 `功能名-task.md` 的设计依据；`功能名-task.md` 用于跟踪执行拆分、进度状态、阻塞项和下一步。

#### B2. 方案设计与实施计划

在设计基线上继续展开方案设计与实施计划，按需裁剪但默认覆盖：

- 背景
- 功能清单
- 各模块详细设计
- 业务流程
- 接口与数据影响
- 边界条件与失败处理
- 验收清单
- 非功能需求
- 测试方案与验收用例

要求：

- 需求框架和实施计划是同一套设计输出的两个视角，不要在两个章节里重复堆砌相同内容。
- 若某个实现细节只影响执行顺序而不改变设计结论，更新实施计划和任务文档即可；若会影响设计基线，必须回写设计文档。

#### B3. 任务拆分与文档落库

进入阶段 B 后，若用户未明确禁止，默认除聊天输出外还需生成两份 Markdown 文档：

- `功能名-desgin.md`：记录需求定义、设计边界、代码落点、实施计划，是 `功能名-task.md` 的设计依据
- `功能名-task.md`：记录执行拆分、进度状态、阻塞项和下一步，必须与 `功能名-desgin.md` 保持同步

若需求目标、边界、接口、字段、数据结构、实现顺序、验收口径、任务优先级或范围发生变动，先更新 `功能名-desgin.md`，再同步 `功能名-task.md`；若执行中发现变动已经影响设计结论，也必须回写设计文档。

生成 `功能名-task.md` 时，任务拆分顺序必须遵循：

1. 数据库 DDL / DML 脚本
2. 代码编写：涉及接口入参出参字段的修改，包括新接口、老接口字段调整；此阶段只做接口契约，不实现业务逻辑
3. 代码编写：一个一个实现接口的具体逻辑，按接口或调用链拆成可独立验收的任务
4. 编写测试用例，包括接口用例、核心业务分支、失败分支和必要的回归验证

要求：

- 若某一类任务不涉及，仍保留对应章节并标明“不涉及”。
- 每个任务项必须能被独立勾选，避免把多个接口或多类动作塞进一个任务。
- 每个任务项默认包含添加时间；完成后补完成时间。
- 执行过程中发现新增任务、阻塞项、优先级变化或范围变化时，必须同步更新 `功能名-task.md`。
- 文档默认拆成两份：`功能名-desgin.md` 和 `功能名-task.md`。
- `功能名-desgin.md` 内容至少包含“阶段 A 结论沉淀 + 方案设计与实施计划”。
- `功能名-task.md` 内容至少包含“当前阶段 + 当前优先级 + 当前阻塞项 + 已完成事项 + 建议的下一步 + 维护规则 + 文档联动规则”。
- 文档模板存放在本技能 `templates/` 目录：`feature-desgin-template.md` 和 `feature-task-template.md`。
- 文档保存路径优先级固定为：
  1. `doc-wiki` 对应业务目录
  2. 当前工作目录下的 `doc/`
  3. 若以上都无法明确判断，再询问用户
- 若同时存在多个明显候选目录，优先选择 `doc-wiki`。
- 若目标目录可从上下文明确推断但不存在，可自动创建子目录；只有归属不明确时才询问用户。
- 若已有同主题文档，优先更新原文档，不默认新建重复文档。
- 若需求讨论已经收敛但文档路径仍不明确，先询问文档归属，不把最终计划只留在聊天里。

## 文档联动
- 设计文档：`功能名-desgin.md`，用于沉淀需求定义、设计边界、代码落点和实施计划
- 任务文档：`功能名-task.md`，用于跟踪执行拆分、进度状态、阻塞项和下一步
- 需求变动处理：先更新 `功能名-desgin.md`，再同步 `功能名-task.md`

## 强约束

1. 即使用户要求“直接开始写代码”，也先完成最小必要澄清与验证闭环。
2. 未完成信息收敛前，不得输出最终计划。
3. 用户明确要求结合代码分析时，必须先看代码再追问；只要当前任务存在可访问项目上下文，默认先查已有代码，而不是先问“要不要看代码”。
4. 不得把能从代码直接验证的问题优先抛给用户。
5. 每次输出都必须显式标明当前阶段；处于澄清与验证时标明阶段 A，处于计划产出时标明阶段 B。
6. 本技能全程禁止执行编码或业务代码修改；仅允许在阶段 B 保存规划类 Markdown 文档。
7. 用户未明确禁止时，阶段 B 默认要保存 Markdown 设计文档和任务文档，不要只把计划留在聊天里。
8. 文档路径优先级固定为：`doc-wiki` 对应业务目录 -> 当前工作目录 `doc/` -> 无法判断时再询问用户。
9. 阶段 A 默认结构为：目标重述与收敛目标 -> 代码仓库验证与术语映射 -> 风险与数据流验证 -> 决策性澄清问题 -> 必要时输出调研表。
10. 阶段 B 默认结构为：阶段 A 结论沉淀 -> 方案设计与实施计划 -> 任务拆分与文档落库。
11. 阶段 B 文档默认命名为 `功能名-desgin.md` 和 `功能名-task.md`，其中 `desgin` 拼写保持不纠正。
12. 任务拆分顺序固定为：数据库 DDL/DML 脚本 -> 接口入参出参字段修改且不实现逻辑 -> 逐个接口实现具体逻辑 -> 编写测试用例。
13. 任何需求目标、边界、接口、字段、数据结构、实现顺序、验收口径、任务优先级或范围变动，都必须优先同步相关设计文档和任务文档。
